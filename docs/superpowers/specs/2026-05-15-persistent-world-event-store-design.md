# Persistent World Event Store Design

Date: 2026-05-15

## Status

Approved design for the next backend persistence slice. This follows the implemented authoritative command/event boundary and turns accepted world events from process-local history into durable append-only records.

## Goal

Persist accepted world mutation events durably after server validation and before websocket broadcast, without adding auth, frontend command UI, full replay recovery, or database-backed chunk snapshots in this slice.

## Context

The current Rust backend owns the only mutation path:

- `POST /commands` accepts versioned local-development commands.
- `SimulationRuntime::apply_client_command` validates commands and mutates loaded hot chunk state.
- Accepted `set_tile_kind` commands append `WorldEventDto::TileKindSet` into `InMemoryWorldEventStore`.
- Accepted events are broadcast as `ServerMessageDto::WorldEvent` over `/ws`.
- No-op tile changes are rejected and do not append or broadcast events.
- Chunk snapshots are written to `InMemoryChunkSnapshotStore` by the snapshot loop.

The architecture target in `docs/superpowers/specs/2026-05-14-abutown-simulation-architecture-v2-design.md` requires Rust authority, validated indirect actions, append-only domain/tile mutation events, periodic chunk snapshots, and Supabase/Postgres as the durable source of record. This slice addresses the event-history part only.

## Non-Goals

- No auth, player accounts, permissions, or Row Level Security.
- No command idempotency behavior beyond preserving `command_id` in durable rows.
- No browser command UI.
- No Supabase Realtime integration.
- No durable chunk snapshot table.
- No hot-state recovery or replay application on server startup.
- No dynamic chunk loading or chunk unload persistence.
- No per-frame or per-tick database writes.

## Design

Accepted world events must pass through an explicit persistence boundary. The runtime may still use an in-memory store in tests, but production-shaped code must depend on a store interface instead of a concrete vector.

The event write order is:

1. Validate command.
2. Plan the loaded hot-state mutation without changing hot state.
3. Build a `WorldEventDto` with the reserved event id and planned authoritative version.
4. Append the event to the configured event store.
5. Apply the planned mutation to loaded hot state.
6. Return `accepted` and broadcast the event.

If the event store append fails, the command must not change hot state, must not be reported as accepted, and must not be broadcast. This requires splitting the current loaded-chunk mutation boundary into validation/planning and application steps. Applying a planned mutation after a successful append should be infallible under the runtime lock; if it fails, that is an internal invariant violation rather than a user-facing validation rejection.

## Store Contract

`sim-core` should expose a small event-store contract shaped around the current DTOs:

- append one `WorldEventDto`,
- return append failures as typed errors,
- list events for tests and diagnostics,
- report event count.

The existing `InMemoryWorldEventStore` remains the default test/local store. The contract should make a persistent adapter possible without forcing the runtime to know about SQL, Supabase, or transport-specific errors.

## Durable Record Shape

The durable event record should preserve both queryable columns and the complete event payload:

- `event_id`: stable event identifier generated by the Rust runtime.
- `world_id`: world identifier.
- `command_id`: client command identifier that caused the event.
- `event_type`: current value `tile_kind_set`.
- `tick`: runtime tick at event creation.
- `version`: authoritative version at event creation.
- `payload`: full serialized `WorldEventDto` JSON.
- `created_at`: database insertion timestamp.

The table should use a uniqueness constraint on `event_id`. It may also index `(world_id, version)` and `(world_id, tick)` so later replay/recovery and admin queries can read ordered history efficiently.

`command_id` should be stored but not made globally unique in this slice. Idempotency semantics need their own design because future clients may retry commands, and rejection/acceptance replay rules must be explicit.

## Persistence Adapter

The first persistent adapter should target Postgres-compatible storage because Supabase is Postgres. The adapter should live in `sim-server` or a focused backend infrastructure module, not in `abutown-protocol`.

Configuration should be explicit:

- default local runtime uses the in-memory event store,
- if a database URL is configured, server startup creates a Postgres event store,
- startup should fail fast if the configured persistent store cannot initialize.

The adapter must serialize `WorldEventDto` through `serde_json` and insert queryable columns derived from the event. It should not subscribe to Supabase Realtime or expose database credentials to the browser.

## Error Handling

Event-store failures are command-acceptance failures. The HTTP handler should return an error response that is distinguishable from validation rejections. The initial code can use a server-side rejection code such as `event_store_unavailable` or an HTTP 500 response, but it must not return an accepted command when the append failed.

Validation errors remain existing command rejections:

- `protocol_mismatch`
- `wrong_world`
- `chunk_not_loaded`
- `tile_out_of_bounds`
- `no_state_change`

No-op commands must still reject before any store append attempt.

## Testing Strategy

Tests must cover the boundary at three levels:

- `sim-core`: the event-store contract preserves append order and reports typed append failures through a failing test store.
- `sim-server` runtime: accepted commands append exactly one event; no-op commands append none; configured append failure does not return accepted and does not mutate hot chunk state.
- `sim-server` HTTP/websocket: accepted commands still broadcast world events; failed append does not broadcast an event.

The implementation plan should keep tests deterministic and avoid requiring a live Supabase project. If a Postgres adapter is implemented in this slice, its integration tests must be opt-in behind an environment variable so normal `cargo test --manifest-path backend/Cargo.toml --workspace` remains local and reliable.

## Documentation

`backend/README.md` should describe the new boundary:

- accepted events are append-only and durable when a persistent store is configured,
- local development defaults to in-memory unless database configuration is present,
- replay/recovery and snapshot persistence are still later slices.

The command-event plan should remain the source of truth for the current local command behavior; this spec only supersedes the “events are in-memory only” limitation.

## Success Criteria

- The runtime no longer hard-codes `InMemoryWorldEventStore`.
- Accepted command events pass through a store abstraction before HTTP acceptance and websocket broadcast.
- Store append failure leaves the loaded chunk snapshot unchanged.
- The local in-memory path remains green.
- A persistent Postgres/Supabase-compatible event-record design exists with migration-ready fields.
- Normal backend verification passes without a live database.
- The spec does not add player auth, frontend mutation UI, recovery replay, or durable snapshots.
