# Procedural Population & Shared Path Network Design

**Date:** 2026-05-16
**Status:** Approved for planning — Phase 3 of the Million-Agent Roadmap.
**Roadmap:** `docs/superpowers/specs/2026-05-16-million-agent-roadmap-design.md`
**Predecessors:** Phase 1 (Visible Backend Mobility), Phase 2 (Frame Interpolation)
**Doctrine:** `docs/literature/agent-simulation/README.md` (SUMO + MATSim agent-centric mobility model)

---

## Goal

Backend simulates **~1000 visible mobility entities** distributed across the full 256×256 city per the existing Zurich road and pedestrian network, with all entities expressed in the SUMO/MATSim-grounded **agent-as-person + vehicle-as-container** model that `agent-mobility-foundation` already established. The browser canvas shows the whole city alive: walking pedestrians on sidewalks across all 64 chunks plus cars on arterials, each car visibly driven by an agent occupant.

## Doctrine Realignment (why Phase 3 is also a corrective slice)

Phase 1's `RoadVehicleWorld` was an animation-only shortcut that violated the foundation doctrine. The doctrine from `agent-mobility-foundation` says:

> *Agents are people with plans and mobility state. Vehicles are separate traffic/transit entities with route, progress, capacity, dwell time, and occupants. Riding is represented by `AgentMobilityState::InVehicle`; the passenger position is derived from the vehicle position.*

Driving a car is just a special case of "riding a vehicle" with capacity 1 — the doctrine already covers it. Phase 1's parallel `RoadVehicleWorld` (records with no occupants, separate persistence, separate protocol) was a shortcut that grew alongside the proper model and is now in the way. Phase 3 retires it.

After this phase, **everything mobile in the world is either an Agent or a Vehicle in `MobilityWorld`**. Visible cars are Vehicles with capacity 1 and a driver-Agent; visible pedestrians are Agents in Walking state; visible trams are Vehicles with capacity N with passenger-Agents.

## Non-Goals

- Realistic traffic AI (intersection priority, congestion). Vehicles follow deterministic routes.
- Per-agent activity-based behavior modeling beyond simple cyclic plans. Future slices can layer it in.
- Player-initiated mobility commands.
- Backend porting of the frontend's procedural city-generation algorithm. The frontend's generator stays the source of truth.
- Phase 4's viewport filter. Phase 3 scales to ~1000 entities which sits comfortably within bandwidth.
- Pak128 sprite catalogue expansion.

## Architecture

Three coordinated changes:

1. **Shared city descriptor** generated from the frontend's existing `zurichTransport` build, committed as JSON, consumed by both frontend and backend.

2. **`MobilityWorld` scales via the descriptor.** Seeder reads pedestrian corridors and arterial paths; produces walking agents per pedestrian corridor, car-Vehicles per arterial with one driver-Agent per car. Plans are simple cyclic walk-to-activity loops (or "stay-in-vehicle forever" for drivers in this slice — richer behaviour is a later slice).

3. **`RoadVehicleWorld` removed.** All cars become Vehicles in `MobilityWorld` with `kind: VehicleKind::Car`. Frontend reads `mobilityState.vehicles`, branches sprite choice on `kind`. Postgres migration drops the `road_vehicle_snapshots` table.

## Shared City Descriptor

### Format

JSON, generated by a Node script that imports the existing TS city build:

```json
{
  "version": 1,
  "world_id": "zurich-river-city-v1",
  "chunk_size": 32,
  "world_tiles": { "width": 256, "height": 256 },
  "arterial_paths": [
    [ { "x": 120, "y": 128 }, { "x": 121, "y": 128 }, ... ],
    ...
  ],
  "pedestrian_corridors": [
    [ { "x": 121, "y": 130 }, ... ],
    ...
  ]
}
```

Arterial paths are the same `zurichTransport.arterialPaths` the frontend uses today. Pedestrian corridors are the `buildPedestrianCorridors` output, pre-filtered by demand (urban-segments-only).

### Generation

- `scripts/generate-city-network.mjs` — Node ESM script.
- Imports `zurichWorld`, `zurichTransport`, `buildPedestrianCorridors` from `src/city/` (same modules Vite uses).
- Writes `data/city/zurich-network.json`.
- `npm run generate:city-network` invokes it.
- Output is **committed** to git so backend and CI don't need to run Node.
- The script is deterministic — the world build is deterministic.

### Consumption

- **Frontend**: `import network from '../../data/city/zurich-network.json'` — Vite handles JSON imports natively.
- **Backend**: `sim-core/src/city_network.rs::load_from_path(path: &str) -> CityNetwork` reads via `std::fs::read_to_string` + `serde_json::from_str`. Path comes from env `ABUTOWN_CITY_NETWORK_PATH` (default `data/city/zurich-network.json` relative to the workspace root).

If the file is missing on backend boot the server fails fast with a clear error — same fail-fast policy as missing `DATABASE_URL`.

## Backend Changes

### `MobilityWorld` extensions

`VehicleRecord` gets a `kind` field:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum VehicleKind {
    Car,
    Tram,
}

pub struct VehicleRecord {
    pub id: VehicleId,
    pub kind: VehicleKind,
    pub route_id: RouteId,
    pub link_index: usize,
    pub progress: f32,
    pub speed_per_tick: f32,
    pub capacity: u16,
    pub occupants: Vec<AgentId>,
    pub dwell_ticks_remaining: u16,
}
```

`VehicleMobilityDto` mirrors the new field. Existing tram seeding sets `kind: VehicleKind::Tram`; new car seeding sets `kind: VehicleKind::Car`.

### Seeder

`backend/crates/sim-core/src/mobility/seed.rs::from_network(network: &CityNetwork, density: SeedDensity) -> MobilityWorld`:

```rust
pub struct SeedDensity {
    pub pedestrians_per_corridor: u32,   // default 6 → ~600 walking agents
    pub cars_per_arterial: u32,          // default 4 → ~400 cars + 400 driver-agents
    pub trams_total: u32,                 // default 4 (preserves existing)
}
```

Logic:
1. For each pedestrian corridor, spawn `pedestrians_per_corridor` walking agents distributed by deterministic offset. Each agent gets a cyclic plan: `WalkToActivity → Activity → WalkToActivity → ...` using a derived activity id from the corridor's midpoint chunk. Walking happens on the corridor's links — each corridor's tile-list is registered as a multi-segment link in the route map.
2. For each arterial path, spawn `cars_per_arterial` Vehicles with `kind: Car`, `capacity: 1`, `speed_per_tick: 0.5`, looped route over the path tiles. For each car spawn one driver-agent with `state: InVehicle { vehicle_id, seat_index: 0 }` and an indefinite "stay riding" plan stage.
3. Preserve the existing 4 trams (or recreate them from a small fixed transit-route table that lives alongside this seeder).

The existing `seed::initial_world()` is renamed to `seed::tiny_world()` and kept as a test-only fixture so the existing 20-agent unit tests keep working.

### `AgentMobilityState`

No new variants in this slice. `InVehicle { vehicle_id, seat_index }` already serves both passenger and driver semantics. The 400 driver-agents stay in `InVehicle` indefinitely; their position is derived from the vehicle's `world_coord`.

### Multi-segment paths inside links

Today `LinkGeometry { start, end }` represents one straight segment. Real corridors are multi-segment (zig-zag through streets). Two options were considered:

- **Option A — extend LinkGeometry to a polyline** of `points: Vec<(f32, f32)>`, with `world_coord_for_progress` walking the polyline by arc length.
- **Option B — decompose each multi-segment corridor into many short atomic LinkGeometry segments**, give each its own `link_id`, advance the agent through them as plan stages.

**Choice: Option A**, extend `LinkGeometry` to a polyline. Reasons: keeps plan-stage count down (one corridor = one link, not 50 atomic links); matches how the frontend already represents corridors; the arc-length walking math is 15 lines.

Updated `LinkGeometry`:

```rust
pub struct LinkGeometry {
    pub points: Vec<(f32, f32)>,  // ≥ 2 points
}
```

`world_coord_at_progress(progress: f32) -> (f32, f32)` walks the polyline by cumulative arc length, clamped to [0, 1]. Direction returned at any point is the local segment direction.

Hardcoded test-world links (used by `tiny_world`) become `points: vec![start, end]` — a 2-point polyline is identical to the old straight segment.

### Persistence

`MobilityWorld` already persists via `mobility_snapshots`. With ~1000 agents + ~400 vehicles in the JSONB payload the row grows to ~500-800 KB. UPSERT every 5 s. Comfortable for Phase 3. Phase 7's per-chunk partitioning addresses the size at scale.

No schema change other than the `kind` field on vehicle records inside the JSONB payload. Pre-existing snapshots without `kind` won't deserialise — acceptable: Phase 3 is the first deployment with cars in mobility, and existing tram-only rows from Phase 2 will fail to load. Handled via fail-fast hydration error; operator drops/truncates `mobility_snapshots` before deploy (one-time migration step documented in the plan).

## Frontend Changes

### Drawables projector

`carsFromMobilityState(state, sprites, now, tickPeriodMs)` switches its source from `state.roadVehicles` to `state.vehicles` filtered by `kind === 'car'`. Returns `BackendCar[]` as before. No call-site change in `main.ts`.

`pedestriansFromMobilityState` continues to read from `state.agents`. Agents in `InVehicle` state are filtered out (rendered as part of their vehicle, not as a separate pedestrian).

Trams are visible as vehicles with `kind === 'tram'` — they get a tram sprite via a new branch in the projector, or via the same `vehicleSprites` catalog matched by `sprite_key`. For simplicity: trams use the same vehicle sprite catalog; `sprite_key` distinguishes (`tram:0`, `car:0`).

### Mobility state

`MobilityOverlayState.roadVehicles` field is removed. `state.vehicles` (already there, was empty for cars before) now carries cars + trams.

The interpolation buffer for vehicles (`InterpolatedEntry<VehicleMobilityDto>`) already exists from Phase 2.

### Removal

`src/backend/roadVehicleProtocol.ts`, `src/backend/roadVehicleState.ts`, related tests removed.

## Scale & Bandwidth

- ~600 walking agents + ~400 driver-agents = ~1000 agent DTOs per delta.
- ~400 car-vehicles + ~4 tram-vehicles = ~404 vehicle DTOs.
- Total per delta: ~1400 DTOs × ~200 B JSON = ~280 KB. At 10 Hz that's ~2.8 MB/sec/client.

That's above my earlier 200 KB/sec estimate because driver-agents are full agent DTOs even though their position is vehicle-derived. Two mitigations available:

- **Mitigation A**: filter `InVehicle` agents out of delta broadcasts. Frontend derives them from vehicle state. Saves ~400 agent DTOs/tick → drops to ~600 KB/sec.
- **Mitigation B**: omit `world_coord` from `InVehicle` agents (server sends placeholder; frontend doesn't read it for these agents). Saves bytes but not entry count.

Adopt **Mitigation A** as part of Phase 3 — it's clean and aligns with the doctrine ("passenger position is derived from the vehicle position"). Server-side: skip changed-agent inclusion when an agent's `state` is `InVehicle`. Frontend: when computing drawable lists, if an agent is in `mobilityState.agents` AND in `state: InVehicle`, skip the pedestrian projection.

Result: ~600 walking-agent DTOs + ~404 vehicle DTOs per delta = ~200 KB delta JSON → ~2 MB/sec/client. Still higher than ideal; lives within dev budget; Phase 4 viewport filter addresses it for real-world deployment.

## Failure Modes

- **Descriptor JSON missing** at backend boot → fail-fast with explicit error pointing to `ABUTOWN_CITY_NETWORK_PATH`.
- **Descriptor schema mismatch** (e.g. older version) → fail-fast on serde deserialization with descriptive error.
- **Old `mobility_snapshots` row without `kind`** → fail-fast hydration error. Operator drops the row pre-deploy. Documented in the migration plan.
- **Frontend Vite build with missing descriptor file** → build fails (Vite errors on import). Forces the generator to run.
- **`InVehicle` agent with stale vehicle_id** (vehicle was removed) → skip rendering and log once per startup.

## Risks

| Risk | Mitigation |
|---|---|
| Polyline-based LinkGeometry math has off-by-one or normalisation bugs | Unit tests at known progresses; visual review in dev. |
| Existing `mobility_snapshots` rows without `kind` cause hydration failure | Documented one-time pre-deploy step: `DELETE FROM mobility_snapshots`. |
| 1000 agent plans add tick-time cost | Plans are simple cyclic; agent count is 50× today, tick cost in HashMap iter is still microseconds. Profile if needed in Phase 5 (ECS migration). |
| Generator script drifts from frontend's runtime view | CI runs the script; verifies the committed JSON is reproducible. Add as Phase-8 gate later if not in this slice. |
| Vite JSON import inflates bundle | ~50–150 KB JSON, Vite handles imports lazily if dynamic, statically if `import x from`. Static import OK at this size. |
| Loss of `RoadVehicleSnapshotStoreError` type breaks external consumers | None outside this repo. Safe to remove. |

## Resolved Questions

- **Scale**: ~1000 visible entities (~600 walking + ~400 driving + 4 trams). Phase 4 unlocks 10k+.
- **Architectural shape**: Agent + Vehicle per `agent-mobility-foundation` doctrine. Cars are Vehicles with `kind: Car`. No new subsystem.
- **Polyline links**: extend `LinkGeometry` to `points: Vec<(f32, f32)>`.
- **`RoadVehicleWorld` fate**: removed in this slice. Postgres table dropped via new migration.
- **Descriptor format**: JSON, generated by Node script, committed to repo.
- **Descriptor consumption**: backend loads via `std::fs` at startup (env-configurable path); frontend imports via Vite JSON loader.
- **Generator determinism**: same TS world build that the frontend uses; assumed deterministic.
- **Driver-agent plans**: stay `InVehicle` indefinitely in this slice. Richer driving plans (parking, switching modes) are a future slice.
- **Trams**: stay as `kind: Tram`, behaviour unchanged.
- **Bandwidth**: filter `InVehicle` agents from delta broadcasts. Driver-agent positions derived client-side from vehicle positions.
- **Old mobility_snapshots rows**: fail-fast hydration; operator drops the table once before deploy.
